## О проекте

Contract Risk Analyzer — production-ready AI-система для автоматизированного анализа рисков в международных контрактах на морские грузоперевозки зерна. Разработана для **ТД «Зерно Заволжья»** — одного из лидеров по экспорту зерна в России, ежегодно отгружающего сотни тысяч тонн зерновых культур через порты Юга России.

Трейдеры и юристы компании ежедневно обрабатывают десятки контрактов на куплю-продажу и перевозку зерновых грузов по международным маршрутам. Каждый контракт содержит десятки юридических формулировок, напрямую влияющих на финансовые риски сделки: условия оплаты, демерредж, лайтайм, качество груза, фрахт, страхование, арбитражные оговорки по международным стандартам зерновой торговли. До внедрения системы анализ одного контракта занимал у специалистов 2–3 часа и был подвержен человеческим ошибкам — пропущенная формулировка могла обернуться убытками в сотни тысяч долларов.

Система полностью автоматизирует процесс: сотрудник загружает контракт через Telegram-бота, AI-модель анализирует текст по 111 рисковым и обязательным формулировкам с учётом контекста сделки — тип продукта, страна назначения, базис поставки Incoterms, порт отгрузки — и формирует структурированный отчёт с уровнями риска, рекомендациями и точными цитатами из договора. Результат загружается в виде Excel-файла, готового для передачи между отделами.

## Задача

Создать AI-систему, способную заменить ручной анализ контрактов юристами и трейдерами. Провести полный ML-цикл: от исследования предметной области и бенчмаркинга моделей через обучение собственной LLM до production-деплоя. Обеспечить высокую точность анализа через двухступенчатую верификацию с применением паттерна LLM-as-a-Judge, а также заложить возможность перехода с коммерческого API на self-hosted модель для обеспечения конфиденциальности контрактных данных заказчика.

## Архитектура

Система спроектирована как микросервисная архитектура из трёх независимых компонентов, объединённых общей Docker-сетью: аналитическое API-ядро, Telegram-бот для пользовательского взаимодействия и исследовательский модуль для R&D и обучения моделей.

### Аналитическое API

Центральный компонент системы — асинхронный REST API, реализующий основной интеллектуальный пайплайн обработки контрактов. Архитектура построена на двухступенчатом LLM-пайплайне с применением паттерна **LLM-as-a-Judge**. На первом этапе основная языковая модель анализирует контракт по каждой из 111 рисковых и обязательных формулировок, генерируя chain-of-thought рассуждения, рекомендации и точные цитаты из текста договора. На втором этапе независимая модель-верификатор выступает в роли «судьи» — она получает исходный контракт, первоначальный анализ и выносит собственное суждение, подтверждая или опровергая решение первой модели. Этот подход значительно снижает количество как ложноположительных, так и ложноотрицательных срабатываний, что критически важно в юридическом контексте.

Каждая формулировка проверяется не изолированно, а контекстно-зависимо — с учётом четырёх параметров конкретной сделки: типа товара, страны назначения, базиса поставки и порта отгрузки. Это позволяет системе корректно интерпретировать неоднозначные формулировки: например, ограничение на использование определённого типа судов является риском только при конкретном базисе поставки и конкретных портах.

Обработка контрактов выполняется асинхронно в фоновом режиме с отслеживанием прогресса. Результат анализа включает по каждой формулировке: факт наличия, уровень риска, цепочку рассуждений модели, рекомендацию, цитату из договора и результат верификации.

### Telegram-бот

Пользовательский интерфейс для сотрудников компании — Telegram-бот, обеспечивающий удобный доступ к аналитическому API без необходимости работы с техническими инструментами. Сотрудник загружает документ контракта, бот отправляет его на анализ и в реальном времени отображает прогресс обработки. По завершении формируется структурированный Excel-отчёт со всеми результатами анализа, который можно сразу передать коллегам из юридического, торгового или логистического отдела.

### Research & Model Training

Исследовательский модуль, в рамках которого был проведён полный R&D-цикл — от первых экспериментов до обучения production-ready модели.

**Бенчмаркинг и выбор архитектуры.** На первом этапе было проведено систематическое сравнение нескольких подходов к решению задачи. Тестировались коммерческие языковые модели, открытые LLM различных масштабов, а также RAG-решения — как SaaS-платформы, так и собственная реализация на основе графового RAG с мультиязычными эмбеддингами и гибридным поиском. Все подходы оценивались по метрике balanced accuracy на наборе из реальных контрактов заказчика с экспертной разметкой от специалистов компании. Это позволило обоснованно выбрать оптимальную архитектуру: двухступенчатый LLM-пайплайн с верификацией показал наилучшее соотношение точности и надёжности.

**Генерация обучающих данных.** Для обучения собственной модели был разработан пайплайн синтетической генерации данных: 59 реальных контрактов заказчика были обработаны по каждой из 111 формулировок, что дало корпус из почти 6000 обучающих примеров. Каждый пример включал полный контекст — системную инструкцию, описание рисковой формулировки, отраслевой словарь и текст контракта — в связке со структурированным JSON-ответом.

**Fine-tuning собственной LLM.** На основе сгенерированного корпуса была проведена серия экспериментов по дообучению открытых языковых моделей различного масштаба — от компактных (менее 1 млрд параметров) до крупных (32 млрд). Применялся метод QLoRA для параметрически-эффективного дообучения с 4-битной квантизацией, что позволило обучать модели с десятками миллиардов параметров на доступном GPU-оборудовании. Обучение велось исключительно на ответной части примеров, использовалось масштабирование контекстного окна для работы с длинными контрактами (до 18 000 токенов). Итоговая модель обеспечила качество, сопоставимое с коммерческим API, при полном контроле над данными и стоимостью инференса.

## Стек технологий

**Backend:** Python 3.12, FastAPI, LangChain (цепочки, JsonOutputParser, @chain), Pydantic v2, aiogram 3, aiohttp, openpyxl

**ML/AI:** OpenAI GPT-4o, Unsloth (2x ускорение fine-tuning), QLoRA (4bit NF4, rank 32–256), TRL (SFTTrainer), Transformers, Qwen2.5, Distilabel, LightRAG, Ollama, SetFit, PyTorch + CUDA 12.1

**Данные:** python-docx, markitdown, Pandas, scikit-learn (balanced_accuracy_score), HuggingFace Hub, Git LFS

**Инфраструктура:** Docker Compose, GitLab CI/CD, Makefile, pre-commit hooks (Ruff), HTTP Basic Auth

## Ключевые достижения

- **Сокращение времени обработки контракта в 12–15 раз** — анализ одного контракта, занимавший у юриста или трейдера 2–3 часа ручной работы, теперь выполняется системой за 10–15 минут. Это позволило юридическому отделу обрабатывать до 20–30 контрактов в день вместо прежних 3–5 и высвободить специалистов для работы над стратегическими задачами
- **Повышение качества и полноты проверки** — система проверяет каждый контракт по всем 111 рисковым формулировкам без исключения, тогда как при ручном анализе специалисты из-за нехватки времени нередко фокусировались лишь на ключевых пунктах. Двухступенчатая верификация через паттерн LLM-as-a-Judge существенно снизила количество пропущенных рисков и ложных срабатываний
- **Стандартизация процесса между отделами** — единый формат Excel-отчётов обеспечил прозрачное взаимодействие между юридическим, торговым и логистическим отделами. Каждый специалист получает одинаково структурированную информацию с уровнями риска, рекомендациями и цитатами из договора, что устранило разночтения и ускорило согласование сделок
- **Полный ML-цикл** — от R&D и систематического бенчмаркинга шести подходов через генерацию обучающих данных и fine-tuning до production-деплоя. Каждое архитектурное решение подтверждено метриками на реальных контрактах заказчика
- **LLM-as-a-Judge** — реализован паттерн двухступенчатой верификации, при котором независимая модель-судья оценивает решения основной модели и может их пересмотреть. Этот подход, заимствованный из практики оценки качества LLM-систем, обеспечивает дополнительный уровень надёжности, критически важный при анализе юридических документов с высокой ценой ошибки
- **Domain-specific NLP** — 111 рисковых формулировок с 5 уровнями риска, словарь из 200+ отраслевых терминов международной зерновой торговли, контекстно-зависимая проверка по 4 параметрам каждой сделки
- **Fine-tuning для обеспечения конфиденциальности** — обучение собственной модели позволило перейти с коммерческого API на self-hosted решение, обеспечив полный контроль над контрактными данными заказчика и прогнозируемую стоимость инференса
