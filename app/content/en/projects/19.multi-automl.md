## About the Project

In the work of a Security Operations Center (SOC), analysts regularly encounter tasks that lend themselves well to machine learning -- classification, pattern discovery, forecasting. However, most SOC teams lack dedicated ML engineers: security analysts have excellent expertise in threats and data handling but do not write ML pipelines in Python.

MultiAutoML is an internal corporate platform that solves this problem. The service enables SOC analysts to independently train machine learning models on their own data without writing a single line of code -- through a browser interface. Simply upload a tabular file, specify the target variable, select the task type, and click a button. The system automatically performs feature preprocessing, algorithm and hyperparameter selection, trains the model, and presents results with interactive charts and a metrics table.

The platform was deployed in production and used by a team of 5-15 SOC analysts for solving applied information security tasks.

## Objective

Lower the barrier to machine learning for information security analysts. Instead of engaging an ML engineer or mastering the Python ecosystem, a SOC specialist should be able to independently test a hypothesis on data -- upload a dataset, receive a trained model with quality assessment, and make a decision about its applicability. The platform must support multiple task types, allow comparing different model configurations, and be easily deployable within internal infrastructure.

## Features

The platform covers three main types of machine learning tasks. Binary classification -- for "yes/no" tasks with a full set of metrics: F1, Accuracy, ROC-AUC, Precision, Recall, PR-AUC. Multiclass classification -- for separating data into multiple categories with weighted metrics. Regression -- for numerical predictions with MAE, MSE, R-squared, MAPE, and SMAPE metrics.

Data handling is flexibly configurable: you can upload a single file with automatic train/test splitting, or provide pre-split train/test files separately. The analyst selects the target variable from a list of columns, can exclude irrelevant features, and set a training timeout -- from 10 seconds to 2 hours depending on data volume.

Training results are presented clearly: a summary metrics table for model comparison, interactive confusion matrix charts, ROC-AUC and Precision-Recall curves with zoom and detailed analysis capabilities. For quick platform onboarding, three demo datasets are built in -- one for each task type -- along with a step-by-step user guide directly in the interface.

## Architecture

### Multi-Model Orchestration

The key architectural decision is the multi-model orchestrator pattern. Instead of training a single model, the system allows simultaneously launching multiple AutoML models with different configurations and comparing their results on the same data. The orchestrator sequentially trains each model with a progress bar display, collects predictions and metrics, and then generates a comparative results table. This approach allows the analyst to make an informed choice rather than a blind decision, selecting the best model based on objective metrics.

### Unified Interface on Top of AutoML

Inside the platform, an abstraction layer wraps the LightAutoML framework (developed by Sber) into a standard scikit-learn-like interface -- training, prediction, evaluation. This layer smooths out framework API quirks: it automatically handles different output formats for binary and multiclass classification, correctly manages feature and target variable roles. The architecture is designed so that adding a new AutoML backend requires no changes to the interface or orchestrator logic -- only implementing an adapter with the same contract.

### Containerization

The application is fully containerized via Docker and Docker Compose. The entire stack -- from the ML framework to the web interface -- is deployed with a single command on port 80. This ensures an identical environment for all analysts and eliminates dependency compatibility issues, which is critical for a tool used by a non-technical team.

## Key Achievements

- **Democratizing ML in SOC** -- information security analysts gained the ability to independently train and evaluate models without engaging ML engineers, reducing the hypothesis testing cycle from days to minutes
- **Increased team efficiency** -- the tool was used by a team of 5-15 SOC analysts in production, enabling each specialist to independently solve applied ML tasks
- **Reduced dependency on ML expertise** -- for standard classification and regression tasks, the SOC team no longer depended on external ML specialists, freeing their time for more complex research tasks
- **Transparent model comparison** -- multi-model orchestration enables objective evaluation of multiple configurations on the same data, eliminating subjective "best model" selection
- **Extensible architecture** -- modular design with an abstraction layer on top of AutoML allows connecting new ML backends without changing the application's interface or logic
- **Minimal deployment threshold** -- full containerization and a Russian-language interface with built-in documentation ensured rapid adoption without additional user training
