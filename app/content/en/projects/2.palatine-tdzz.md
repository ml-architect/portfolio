## About the Project

Contract Risk Analyzer is a production-ready AI system for automated risk analysis in international maritime grain shipping contracts. Developed for **TD Zerno Zavolzhya** — one of Russia's leading grain exporters, annually shipping hundreds of thousands of tons of grain through the ports of Southern Russia.

The company's traders and lawyers process dozens of grain purchase and shipping contracts daily across international routes. Each contract contains dozens of legal formulations that directly affect the financial risks of the deal: payment terms, demurrage, laytime, cargo quality, freight, insurance, and arbitration clauses under international grain trade standards. Before the system was deployed, analyzing a single contract took specialists 2–3 hours and was prone to human error — a missed formulation could result in losses of hundreds of thousands of dollars.

The system fully automates the process: an employee uploads a contract via a Telegram bot, the AI model analyzes the text against 111 risk and mandatory phrases considering the deal context — product type, destination country, Incoterms delivery basis, shipping port — and generates a structured report with risk levels, recommendations, and exact quotes from the contract. The result is delivered as an Excel file ready for sharing between departments.

## Objective

Build an AI system capable of replacing manual contract analysis by lawyers and traders. Execute the full ML cycle: from domain research and model benchmarking through training a custom LLM to production deployment. Ensure high analysis accuracy via two-stage verification using the LLM-as-a-Judge pattern, and establish the capability to transition from a commercial API to a self-hosted model to ensure confidentiality of the client's contract data.

## Architecture

The system is designed as a microservice architecture consisting of three independent components united by a shared Docker network: the analytics API core, a Telegram bot for user interaction, and a research module for R&D and model training.

### Analytics API

The central component of the system is an asynchronous REST API implementing the core intelligent contract processing pipeline. The architecture is built on a two-stage LLM pipeline using the **LLM-as-a-Judge** pattern. In the first stage, the primary language model analyzes the contract against each of the 111 risk and mandatory phrases, generating chain-of-thought reasoning, recommendations, and exact quotes from the contract text. In the second stage, an independent verifier model acts as the "judge" — it receives the original contract, the initial analysis, and renders its own judgment, confirming or overturning the first model's decision. This approach significantly reduces both false positives and false negatives, which is critically important in a legal context.

Each phrase is checked not in isolation but context-dependently — considering four parameters of the specific deal: product type, destination country, delivery basis, and shipping port. This allows the system to correctly interpret ambiguous formulations: for example, a restriction on using a certain vessel type is a risk only under a specific delivery basis and specific ports.

Contract processing is performed asynchronously in the background with progress tracking. The analysis result includes for each phrase: presence status, risk level, model's chain of thought, recommendation, contract quote, and verification result.

### Telegram Bot

The user interface for company employees is a Telegram bot providing convenient access to the analytics API without the need to work with technical tools. An employee uploads a contract document, the bot sends it for analysis and displays processing progress in real time. Upon completion, a structured Excel report with all analysis results is generated, ready to be shared with colleagues from the legal, trading, or logistics departments.

### Research & Model Training

The research module within which the full R&D cycle was conducted — from initial experiments to training a production-ready model.

**Benchmarking and architecture selection.** In the first stage, a systematic comparison of several approaches to solving the problem was conducted. Commercial language models, open-source LLMs of various scales, and RAG solutions were tested — both SaaS platforms and a custom implementation based on graph RAG with multilingual embeddings and hybrid search. All approaches were evaluated using the balanced accuracy metric on a set of real client contracts with expert annotations from company specialists. This allowed for a well-founded selection of the optimal architecture: the two-stage LLM pipeline with verification demonstrated the best balance of accuracy and reliability.

**Training data generation.** To train a custom model, a synthetic data generation pipeline was developed: 59 real client contracts were processed against each of the 111 phrases, yielding a corpus of nearly 6,000 training examples. Each example included full context — system instruction, risk phrase description, industry glossary, and contract text — paired with a structured JSON response.

**Fine-tuning a custom LLM.** Based on the generated corpus, a series of experiments was conducted to fine-tune open-source language models of various scales — from compact (under 1 billion parameters) to large (32 billion). The QLoRA method was used for parameter-efficient fine-tuning with 4-bit quantization, enabling training of models with tens of billions of parameters on accessible GPU hardware. Training was performed exclusively on the response portion of examples, and context window scaling was used to handle long contracts (up to 18,000 tokens). The final model achieved quality comparable to the commercial API while providing full control over data and inference costs.

## Tech Stack

**Backend:** Python 3.12, FastAPI, LangChain (chains, JsonOutputParser, @chain), Pydantic v2, aiogram 3, aiohttp, openpyxl

**ML/AI:** OpenAI GPT-4o, Unsloth (2x fine-tuning speedup), QLoRA (4bit NF4, rank 32–256), TRL (SFTTrainer), Transformers, Qwen2.5, Distilabel, LightRAG, Ollama, SetFit, PyTorch + CUDA 12.1

**Data:** python-docx, markitdown, Pandas, scikit-learn (balanced_accuracy_score), HuggingFace Hub, Git LFS

**Infrastructure:** Docker Compose, GitLab CI/CD, Makefile, pre-commit hooks (Ruff), HTTP Basic Auth

## Key Achievements

- **12–15x reduction in contract processing time** — analysis of a single contract, which previously took a lawyer or trader 2–3 hours of manual work, is now completed by the system in 10–15 minutes. This enabled the legal department to process up to 20–30 contracts per day instead of the previous 3–5, freeing specialists to focus on strategic tasks
- **Improved quality and completeness of review** — the system checks every contract against all 111 risk phrases without exception, whereas during manual analysis, specialists often focused only on key points due to time constraints. Two-stage verification via the LLM-as-a-Judge pattern significantly reduced the number of missed risks and false positives
- **Cross-department process standardization** — a unified Excel report format ensured transparent collaboration between legal, trading, and logistics departments. Every specialist receives identically structured information with risk levels, recommendations, and contract quotes, eliminating discrepancies and accelerating deal approvals
- **Full ML cycle** — from R&D and systematic benchmarking of six approaches through training data generation and fine-tuning to production deployment. Every architectural decision is backed by metrics on real client contracts
- **LLM-as-a-Judge** — implemented a two-stage verification pattern where an independent judge model evaluates the primary model's decisions and can overturn them. This approach, borrowed from LLM system evaluation practices, provides an additional layer of reliability critically important when analyzing legal documents with high error costs
- **Domain-specific NLP** — 111 risk phrases with 5 risk levels, a glossary of 200+ international grain trade industry terms, context-dependent checking across 4 parameters of each deal
- **Fine-tuning for confidentiality** — training a custom model enabled the transition from a commercial API to a self-hosted solution, ensuring full control over the client's contract data and predictable inference costs
