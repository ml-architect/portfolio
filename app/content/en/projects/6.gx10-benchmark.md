## About the Project

DGX Spark Benchmark is a service for comprehensive LLM inference benchmarking on **NVIDIA DGX Spark** — one of the first compact devices based on the Grace Blackwell architecture with 128 GB of unified memory.

The results of this benchmark became the basis for a [Habr article](https://habr.com/ru/articles/994058/) — a detailed analysis of the real-world capabilities of DGX Spark, where NVIDIA's marketing claims were verified through objective measurements.

## Objective

Develop a fully reproducible tool that automatically runs models through a set of realistic scenarios, collects performance metrics, and generates ready-to-use interactive charts for a technical publication.

## How the Service Works

### Multi-Dimensional Testing

The service performs a complete sweep of all parameter combinations: each model is tested on each dataset at every concurrency level (from 1 to 100 simultaneous requests). This allows building a complete picture of model behavior — from responsiveness on single requests to the throughput saturation point.

### Three Load Scenarios

Three datasets were prepared for testing, reflecting typical production scenarios:

- **Short chat** — simulating interactive dialogue with short queries, key metric being time to first token
- **Medium-context RAG** — a typical enterprise knowledge base search task with context of several thousand tokens
- **Long-context RAG** — analysis of large documents with context up to 73,000 tokens, a stress test for the memory subsystem

### Metrics

Four key metrics are collected for each run, broken down by percentiles (mean, median, P99):

- **TTFT** — time to first token, critical for user experience
- **TPOT** — time per output token, determines the "typing" speed
- **Throughput** — overall throughput in tokens per second
- **E2E Latency** — total request processing time

### Quantization Comparison

Each model was tested in several precision formats — BF16, FP8, AWQ 4-bit, and GPTQ-Int4 — which made it possible to identify optimal quantization modes for each scenario. One of the key findings was the MoE architecture in GPTQ-Int4: the quality of a 30-billion-parameter model at the computational cost of a 3-billion-parameter one.

## Automated Visualization

A dedicated visualization module automatically generates interactive Plotly charts from all benchmark runs. Seven types of charts are generated for each model: throughput vs. concurrency, TTFT and TPOT analysis, latency percentile distribution, bar charts, and a combined metrics matrix. For multi-model comparison, the same charts are built with panel breakdowns. Charts are exported to HTML and PNG — these were directly used in the final Habr article.

## Infrastructure

The entire service is containerized: vLLM servers with models and the benchmark orchestrator are launched via Docker Compose. The service automatically waits for inference servers to be ready, runs all tests, and saves structured results in JSON. Reproducing the benchmark on different hardware requires no manual configuration.

## Tech Stack

**Inference:** vLLM (optimized LLM inference engine)

**Data Analysis:** pandas, NumPy

**Visualization:** Plotly (interactive charts), Kaleido (PNG export)

**Infrastructure:** Docker, Docker Compose

## Key Results

- **One of the first public DGX Spark benchmarks** — objective evaluation of a new device, valuable material for the ML community
- **Habr article** — the benchmark results became the basis for a [detailed publication](https://habr.com/ru/articles/994058/) analyzing the real-world capabilities of DGX Spark
- **Plateau effect discovery** — throughput stops growing beyond 50 concurrent requests due to LPDDR5X memory bandwidth limitations (273 GB/s)
- **Automated visualization** — interactive Plotly charts were generated automatically and directly inserted into the article
- **Full reproducibility** — containerized pipeline, launch without manual setup
