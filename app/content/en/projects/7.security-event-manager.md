## About the Project

Security Event Manager (SEM) is an automation platform for SOC centers (Security Operations Center), deployed across 8 federal district centers in Russia. Each instance processes thousands of security events daily from regional SIEM systems, automatically detecting cyberattacks and security incidents.

The system covers the full incident response cycle: continuous analysis of the SIEM event stream, threat detection using ML models and Sigma rules, automatic classification, and escalation of incidents to TheHive. A built-in local LLM helps analysts investigate events and make decisions — without sending any data outside the security perimeter.

## Objective

Before SEM, SOC analysts manually triaged event streams from SIEM, spending most of their time on routine classification and unable to process the full volume of data. A significant number of security incidents went undetected due to human limitations and time constraints.

The goal was to build a platform that automates routine operations — analyzing thousands of events, matching them against detection rules, classifying attack types — and allows analysts to focus on investigating real threats. A critical requirement: all data, including AI assistant interactions, must remain within the organization's security perimeter.

## Architecture

SEM is built as a two-container microservice system with a dedicated AI server.

**Web Container** — FastAPI REST API (17 routers) with a Vue 3 SPA, serving the analyst interface. Async SQLite stores configuration, rules, whitelists, and users. Connects to the PostgreSQL SIEM via psycopg2 for event reading and dashboard analytics. Parallel data collection from 5 sources via `asyncio.gather` ensures fast dashboard loading.

**Back Container** — an autonomous processor running in 30–40 minute cycles. Each cycle it pulls new events from SIEM, runs them through CatBoost models and the Sigma engine, groups them by IP address, assigns severity levels, and creates alerts in TheHive. The processor retrieves its configuration from the Web Container via HTTP API and automatically waits for readiness on startup (retry with backoff, 30 attempts).

**Ollama** — a local LLM server running Qwen3 and DeepSeek-R1 models. Powers the AI assistant for analysts: event interpretation, suspicious activity analysis, documentation lookup. A Git2Markdown converter loads code from repositories directly into LLM context.

### Sigma Rules Engine

A custom engine built from scratch. The recursive descent parser supports full Sigma syntax: AND/OR/NOT logical operators, wildcard patterns, quantifiers ("1 of", "all of"), modifiers (contains, startswith, endswith, re, cidr). Extended with sensor filtering and IDS group support. Covered by 11 test scenarios. Analysts can create, edit, and import YAML rules through the UI.

### ML Detection

A two-level CatBoost system: the first model determines whether an attack occurred; the second classifies the attack type (DNS attacks, trojan activity, cryptomining malware, etc.). Models are loaded at startup and applied to every generated incident. Combined with Sigma rules and the tagging system, ML detection provides multi-layered defense where different mechanisms complement each other.

### Rule and Whitelist Management

The system supports multi-level tagging rules (up to 4 nesting levels with AND/OR logic) for describing complex event chains. Three types of whitelists (sensors, signatures, IP addresses) with expiration timers and automatic notifications prevent legitimate alerts from being suppressed longer than necessary. Rule analytics automatically identifies dead (no matches), noisy (anomalously frequent), and declining (> 50% drop) rules.

## Tech Stack

**Backend:** Python 3.11, FastAPI 0.109, aiosqlite, psycopg2, Alembic, HMAC-SHA256, bcrypt

**Frontend:** Vue 3.4, TypeScript 5.3, Vite, Pinia, Tailwind CSS 3.4, Radix Vue, shadcn/vue

**ML/AI:** CatBoost 1.2.7, Ollama (Qwen3, DeepSeek-R1), pySigma

**Infrastructure:** Docker (multi-stage), Docker Compose, GitLab CI/CD

**Integrations:** TheHive, SIEM PostgreSQL

## Key Achievements

- **3x increase in incident detection** — automated detection via ML and Sigma rules identifies three times more security incidents compared to manual analysis
- **70% reduction in processing time** — analysts spend their time investigating threats, not manually triaging the event stream
- **8 instances across Russia** — deployed in federal district centers, each processing thousands of events daily from regional SIEM systems
- **Near-zero missed threats** — multi-layered detection (ML + Sigma + tagging rules) virtually eliminated situations where significant events went unnoticed
- **Local LLM with zero data leakage** — Ollama-based AI assistant operates within the security perimeter, helping analysts without compromising sensitive data
- **Self-maintaining rule base** — automatic identification of ineffective rules maintains detection quality without manual auditing
