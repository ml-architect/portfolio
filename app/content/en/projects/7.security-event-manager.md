## About the Project

Security Event Manager (SEM) is a SOC automation platform deployed across 8 federal district centers in Russia. Each instance processes thousands of SIEM events daily, covering the full incident response cycle: threat detection via ML and Sigma rules, classification, and incident escalation to TheHive. A local LLM helps analysts investigate events without sending data outside the security perimeter.

## Objective

SOC analysts were manually triaging the event stream from SIEM — most of the time was spent on routine classification, and a significant number of incidents went undetected. A platform was needed to automate analysis, rule matching, and attack classification. A critical requirement was that all data, including the AI assistant, remain within the security perimeter.

## Architecture

A two-container system with a dedicated AI server.

**Web Container** — FastAPI REST API (17 routers) + Vue 3 SPA. Async SQLite for configuration and rules, psycopg2 for reading events from PostgreSQL SIEM. Parallel data collection from 5 sources via `asyncio.gather` for fast dashboard loading.

**Back Container** — an autonomous processor running in 30–40 minute cycles. Pulls events from SIEM, runs them through CatBoost and the Sigma engine, groups by IP, assigns severity, and creates alerts in TheHive. Retrieves configuration from the Web Container via HTTP API (retry with backoff on startup).

**Ollama** — a local LLM server (Qwen3, DeepSeek-R1) for the analyst AI assistant: event interpretation, suspicious activity analysis. A Git2Markdown converter loads code from repositories into LLM context.

### Sigma Rules Engine

A custom engine built from scratch. Recursive descent parser with full Sigma syntax: AND/OR/NOT, wildcard, quantifiers ("1 of", "all of"), modifiers (contains, startswith, endswith, re, cidr). Extended with sensor and IDS group filtering. 11 test scenarios. UI for creating, editing, and importing YAML rules.

### ML Detection

Two-level CatBoost: the first model determines whether an attack occurred, the second classifies the attack type (DNS attacks, trojan activity, cryptomining malware, etc.). Combined with Sigma rules and tagging, it provides multi-layered detection.

### Rule and Whitelist Management

Multi-level tagging rules (up to 4 nesting levels, AND/OR logic) for complex event chains. Three types of whitelists (sensors, signatures, IP addresses) with expiration timers and notifications. Rule analytics automatically identifies dead (no matches), noisy (anomalously frequent), and degrading (> 50% drop) rules.

## Tech Stack

**Backend:** Python 3.11, FastAPI 0.109, aiosqlite, psycopg2, Alembic, HMAC-SHA256, bcrypt

**Frontend:** Vue 3.4, TypeScript 5.3, Vite, Pinia, Tailwind CSS 3.4, Radix Vue, shadcn/vue

**ML/AI:** CatBoost 1.2.7, Ollama (Qwen3, DeepSeek-R1), pySigma

**Infrastructure:** Docker (multi-stage), Docker Compose, GitLab CI/CD

**Integrations:** TheHive, SIEM PostgreSQL

## Key Achievements

- **3x increase in incident detection** — ML + Sigma identify three times more incidents compared to manual analysis
- **70% reduction in processing time** — analysts focus on investigation instead of sorting events
- **8 instances across Russia** — each processes thousands of events daily from regional SIEM systems
- **Multi-layered detection** — ML + Sigma + tagging virtually eliminated missed significant events
- **Local LLM** — Ollama-based AI assistant operates within the security perimeter
- **Self-cleaning rule base** — automatic identification of ineffective rules without manual auditing
