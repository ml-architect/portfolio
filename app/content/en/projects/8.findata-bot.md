## About the Project

Findata BOT is a production-ready fintech system in the form of a Telegram bot, built for investors in the Russian bond market. The system solves a key problem: bond investors need to monitor dozens of issuer companies daily across several disparate platforms — disclosure portals, credit rating agencies, exchange data. This consumes hours of time and creates the risk of missing a critically important event — from a rating change to a default on obligations.

Findata BOT fully automates this process. The system aggregates data from six official sources — e-disclosure.ru, the National Settlement Depository (NSD), credit rating agencies ACRA, NRA, and Expert RA, as well as isin.ru — and delivers personalized notifications based on specific bonds in the user's portfolio. The investor receives only relevant information, filtered by International Securities Identification Numbers (ISIN) and keyword triggers.

The project is fully monetized: subscription plans with a trial period, automatic payment processing with fiscal compliance, subscription management, and administration are all implemented.

## Objective

Build a reliable fintech platform that transforms manual monitoring of disparate platforms into automatic delivery of personalized notifications. The platform must cover all key information sources for bond investors, integrate with brokerage APIs for automatic portfolio linking, ensure financial data security, and support the full business cycle — from user registration to payment and subscription management.

## Features

### Corporate Disclosure Monitoring
The user subscribes to companies of interest, and the system automatically tracks news and events on the disclosure portal. Notifications are filtered by specific bond ISIN codes and trigger phrases — the investor receives only what pertains to their portfolio. Four types of financial reports are tracked separately: charter documents, annual reports, accounting statements, and IFRS reports. When critical events are detected — such as a default on obligations — the system immediately alerts the user.

### Rating Notifications
In parallel, the system monitors three leading Russian credit rating agencies — ACRA, NRA, and Expert RA — as well as the National Settlement Depository. Rating actions and events are delivered in two modes: targeted — matching ISIN codes of bonds in the user's portfolio, and broadcast — for events important to all investors.

### New Bond Monitoring
The system tracks the appearance of new bonds on the market: it collects fresh ISIN codes within a rolling 30-day window and checks their availability for purchase through the broker. As soon as a new bond becomes available, the user receives a notification.

### Brokerage Portfolio Linking
Integration with the T-Bank Invest API enables automatic connection of a brokerage account. The system securely stores the API token in encrypted form and regularly synchronizes the bond portfolio during a nightly window, minimizing the load on the external API. Based on the current portfolio composition, subscriptions to corresponding issuers are automatically created, updated, and deactivated — the investor does not need to manually configure monitoring.

### Monetization
A full-featured subscription model is implemented with several pricing plans: from a free tier with a limit on the number of companies to a premium tier with full access to all features. A trial period is provided for new users, along with automatic reminders about subscription expiration and a smooth transition to the free tier without losing settings.

## Architecture

### Two-Process Model
The system is divided into two independent processes running in parallel. The first handles user interaction via Telegram — command processing, menu navigation, subscription configuration, and account management. The second is a set of background tasks, each running on its own schedule: monitoring data sources, sending notifications, synchronizing portfolios, processing payments, and managing subscriptions. This separation allows the bot interface to remain responsive regardless of the load on background processes.

### Extensible Data Source System
The data source architecture is built on a modular principle with a unified registry. Each source is an independent module implementing a common interface: data retrieval, parsing, event normalization, and notification delivery. Adding a new source comes down to implementing a single module and registering it — the rest of the infrastructure (scheduling, deduplication, delivery) picks it up automatically. This enables rapid expansion of the system's coverage when new platforms appear or existing ones change.

### Subscription Management
Subscriptions are designed with the investor's lifecycle in mind. When a brokerage portfolio is linked, subscriptions are automatically created based on the current bond holdings. When changing plans, subscriptions are not deleted but soft-deactivated — the user can restore them with a single action upon upgrading. During nightly portfolio synchronization, the system detects changes and adjusts subscriptions: adding new ones and deactivating those whose bonds are no longer in the portfolio.

### Two-Level Data Access
A deliberate separation is applied for database operations: declarative schema definitions and migrations are handled through ORM, while all runtime operations use low-level asynchronous queries. This ensures convenient schema evolution and maximum performance for data processing.

## Security and Reliability

Financial data security is one of the system's priorities. API tokens for brokerage accounts and user contact information are stored in the database in encrypted form — even if the database is compromised, an attacker would not gain access to this data. All user inputs — links, identifiers, securities codes — undergo strict validation before processing. Access to administrative functions is restricted by a whitelist.

Several mechanisms are implemented for operational reliability. The deduplication system prevents sending the same notification twice under any scenario — failures, restarts, parallel processing. All background processes support graceful shutdown without data loss. Structured logging is configured with separate logs for each component, automatic rotation, and compression — enabling rapid diagnosis of issues in production. Payment processing is implemented through an official payment gateway with automatic fiscal compliance per 54-FZ, ensuring full regulatory compliance.
