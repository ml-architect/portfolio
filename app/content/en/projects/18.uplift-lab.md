## About the Project

The educational platform of the [RANEPA Intelligent Analysis Laboratory](https://lia.ranepa.ru/) includes courses on data analysis and forecasting, where it is important for students not just to study machine learning theory, but to see in practice how the ML approach outperforms manual analytics. Traditional lecture materials poorly convey this difference -- an interactive tool was needed where students could try their hand at being an analyst, then compare their results with algorithm performance.

I designed and developed Uplift Lab -- an interactive web laboratory that implements the "Analyst vs ML" concept. The user manually segments a marketing campaign audience through a filter system, then sees how much better ML models handle this task. Delta metrics clearly show the difference between manual and algorithmic approaches, building students' understanding of uplift modeling value.

At the foundation of the laboratory lies the classic MineThatData E-Mail Analytics benchmark dataset (Kevin Hillstrom) -- over 42,000 customer records from an email marketing campaign with control and treatment group separation.

## Features

Working with the laboratory begins with exploratory data analysis. Interactive charts show customer distribution by purchase channel, price segment, recency of last purchase, and other characteristics -- this helps students "feel" the data before making segmentation decisions.

Next, the user proceeds to manual audience segmentation. The system offers six filter types: purchase channel, customer type, product category, price segments, geography, and recency of last order. The student chooses who, in their opinion, should receive a promotional email -- effectively solving the same task as a marketer in a real company. At each step, validation works with clear messages, preventing errors when working with filters.

After applying filters, quality metrics for the manual segmentation are calculated -- uplift@k, Qini AUC, weighted average uplift. Then the same metrics are calculated for three ML models, and the student sees the result in delta metric format: absolute difference between their approach and each algorithm with up/down arrows. On Qini and Uplift curve charts, a red marker shows the manual analytics position against ML curves -- a visually compelling demonstration of algorithmic superiority.

## Architecture

### ML Pipeline and Modeling Approaches

Uplift modeling solves a key marketing challenge: determining which customers advertising will have the maximum positive effect on, and which it will not. This allows avoiding spending budget on customers who would buy without advertising, or on those whom advertising only irritates.

For the laboratory, I trained 12 model combinations: three gradient boosting algorithms -- CatBoost, XGBoost, and Random Forest -- in each of four uplift modeling approaches via the scikit-uplift library. The approaches differ in their strategy for working with control and treatment groups: from a single model on combined data to a pair of dependent models using the DDR method. Systematic benchmarking showed that the best result is achieved by CatBoost in the SoloModel approach -- uplift@30% = 0.0776.

CatBoost and XGBoost training was performed with GPU acceleration, which allowed processing the entire experimental plan within a reasonable time. For reproducibility, random seeds are fixed and train/test split indices are saved.

### Instant Response Without Server GPU

Model training is a resource-intensive process requiring a GPU. However, the student-facing web application must work fast and on any hosting. I separated these stages: models are trained offline, and their predictions are saved to CSV files. On startup, the application loads pre-computed results -- this ensures instant interface responsiveness without dependence on server computational resources.

### CI/CD and Deployment

The application is deployed on Hugging Face Spaces with Streamlit SDK. An automatic GitHub Actions pipeline on each push checks file sizes for platform compatibility and synchronizes the repository with Hugging Face Hub. Binary artifacts -- trained models and data -- are stored via Git LFS.
