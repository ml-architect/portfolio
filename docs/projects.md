# Добавление проектов

Руководство по добавлению новых проектов в портфолио.

## Расположение файлов

Проекты хранятся в двух директориях (для каждого языка):

```
app/content/
├── ru/projects/    # Русскоязычные версии
└── en/projects/    # Англоязычные версии
```

## Именование файлов

Формат: `{НОМЕР}.{slug}.md`

- Номер — порядковый (следующий свободный, текущие: 1-21)
- Slug — латиница, lowercase, дефис как разделитель
- Пример: `20.my-new-project.md`

Файл с одинаковым именем должен быть создан в обеих локалях.

## Frontmatter

### Обязательные поля

| Поле | Тип | Описание |
|------|-----|----------|
| `title` | string | Название проекта |
| `description` | string | Краткое описание (1 строка, 20-60 слов). На карточке обрезается до 2 строк |
| `category` | `'commercial'` \| `'research'` | Тип проекта. Влияет на badge и фильтрацию |
| `date` | string | Дата в формате `YYYY-MM-DD` |
| `tags` | string[] | Массив технологий (8-15 штук). **Всегда на английском**, даже в ru-версии |
| `role` | string | Роль в проекте (напр. "Единственный разработчик", "ML-инженер") |

### Опциональные поля

| Поле | Тип | По умолчанию | Описание |
|------|-----|------|----------|
| `featured` | boolean | `false` | Показывать в секции "Избранные проекты" на главной |
| `team` | string | — | Команда или организация |
| `client` | string | — | Заказчик. Для закрытых проектов — `'NDA'` |
| `clientUrl` | string | — | URL сайта заказчика (если публичный) |
| `highlights` | string[] | — | 5-10 ключевых достижений с метриками |
| `github` | string | — | Ссылка на GitHub-репозиторий (для открытых проектов) |

## Ограничения отображения

- **Карточка проекта:** показываются первые 4 тега, остальные как `+N`
- **Избранный проект (главная):** 3-5 тегов в зависимости от варианта карточки
- **Description на карточке:** обрезается до 2 строк (`line-clamp-2`)
- **Description в избранных:** обрезается до 3 строк для main-варианта

## Шаблон: коммерческий проект

```yaml
---
title: 'Название проекта'
description: 'Краткое описание проекта в одну строку (не более 60 слов)'
category: 'commercial'
featured: false
date: '2026-01-15'
tags:
  - Python
  - FastAPI
  - PostgreSQL
  - Docker
  - Vue 3
  - TypeScript
role: 'Full-stack разработчик'
team: 'Название команды'
client: 'Название заказчика'
clientUrl: 'https://example.com'
highlights:
  - 'Первое достижение с конкретной метрикой'
  - 'Второе достижение'
  - 'Третье достижение'
  - 'Четвёртое достижение'
  - 'Пятое достижение'
---
```

### Структура контента коммерческого проекта

```markdown
## О проекте

1-3 абзаца: обзор проекта, ключевые особенности, масштаб.
Описание того, что делает проект и какую проблему решает.

## Задача

Контекст проблемы, бизнес-требования, ограничения.
Что именно нужно было сделать и почему.

## Архитектура

### Компонент 1
Описание первого ключевого компонента системы.

### Компонент 2
Описание второго компонента.

### Компонент 3
Описание третьего компонента.

(3-8 подсекций в зависимости от сложности проекта)

## Стек технологий

**Backend:** Python, FastAPI, Uvicorn, Pydantic
**ML/AI:** PyTorch, Hugging Face Transformers
**Frontend:** Vue 3, TypeScript, Tailwind CSS
**Данные:** PostgreSQL, Redis
**Инфраструктура:** Docker, Docker Compose, GitLab CI/CD, Nginx

## Ключевые достижения

- **Результат 1** — описание достижения с конкретной метрикой
- **Результат 2** — описание достижения
- **Результат 3** — описание достижения
```

**Стиль описания проектов:**
- Описание должно быть компактным и лаконичным — без воды, повторов и дублирования информации между секциями
- Не использовать код, имена классов, функций, переменных и т.п. в описаниях
- Описания должны быть понятны нетехническому читателю — без фрагментов кода, путей к файлам или внутренних идентификаторов
- Технологии упоминать по названию (например «FastAPI», «PostgreSQL»), но не приводить примеры кода или API-вызовов
- Перед использованием названий конкретных ML-моделей (например «YOLOv8», «LLaMA 3», «BERT») — всегда уточнять у автора, допустимо ли их упоминание в описании проекта

**Акценты коммерческих проектов:**
- Масштаб и нагрузка (кол-во пользователей, запросов, данных)
- Бизнес-метрики (ROI, конверсия, экономия)
- Надёжность (uptime, SLA)
- Производительность (латентность, throughput)

**Ключевые достижения должны:**
- Описывать пользу для человека
- Отображать конкретные релевантные проекту метрики

**Ключевые достижения не должны:**
- Быть абстрактными или бесполезными
- Быть банальными, например GPU-ускорение это не достижение

## Шаблон: исследовательский/образовательный проект

```yaml
---
title: 'Название проекта'
description: 'Краткое описание образовательного/исследовательского проекта'
category: 'research'
featured: false
date: '2026-01-15'
tags:
  - Python
  - Streamlit
  - Scikit-learn
  - Pandas
role: 'ML-инженер'
team: 'Название команды или лаборатории'
client: 'Университет или организация'
clientUrl: 'https://university.edu'
highlights:
  - 'Интерактивный формат с визуализациями'
  - 'Пошаговый сценарий обучения'
  - 'Открытый исходный код'
github: 'https://github.com/username/repo'
---
```

### Структура контента исследовательского проекта

```markdown
## О проекте

1-2 абзаца: описание проекта, его образовательная/исследовательская цель.

## Задача

Контекст: какую проблему решает, для какой аудитории.
Образовательные цели или исследовательские гипотезы.

## Функционал

Описание ключевых функций и возможностей:
- Интерактивные элементы
- Визуализации
- Учебные сценарии

## Архитектура

### Модуль 1
Описание первого модуля.

### Модуль 2
Описание второго модуля.

(2-4 подсекции)

## Стек технологий

**Backend:** Python, Streamlit
**ML:** Scikit-learn, Pandas, NumPy
**Деплой:** Hugging Face Spaces, GitHub Actions

## Ключевые достижения

- **Результат 1** — описание с фокусом на методологию
- **Результат 2** — описание с фокусом на образовательную ценность
```

**Акценты исследовательских проектов:**
- Методология и научный подход
- Образовательная ценность и интерактивность
- Инновационность подхода
- Воспроизводимость результатов
- Открытость (GitHub, документация)

## Различия между типами проектов

| Аспект | Коммерческий | Исследовательский |
|--------|-------------|-------------------|
| `category` | `'commercial'` | `'research'` |
| `featured` | Может быть `true` | Обычно `false` |
| `github` | Редко (NDA) | Обычно есть |
| `client` | Коммерческий заказчик или 'NDA' | Университет, лаборатория |
| Секция «Функционал» | Нет | Да (опционально) |
| Highlights | Метрики масштаба и бизнеса | Методология и образовательная ценность |
| Объём контента | 80-120 строк | 60-80 строк |
| Подсекции архитектуры | 4-8 | 2-4 |
| Платформа деплоя | Docker, приватная инфраструктура | Hugging Face Spaces, GitHub Pages |

## Примеры тегов по категориям

**Backend:** Python, FastAPI, Django, Flask, Uvicorn, Pydantic, Celery
**ML/AI:** PyTorch, TensorFlow, Scikit-learn, Hugging Face Transformers, LLM, RAG, BERT, CatBoost, Ollama
**Frontend:** Vue 3, TypeScript, Tailwind CSS, Pinia, Streamlit, Nuxt 3
**Данные:** PostgreSQL, Redis, MongoDB, ClickHouse, Elasticsearch, Pandas
**Инфраструктура:** Docker, Docker Compose, Kubernetes, GitLab CI/CD, GitHub Actions, Nginx, Traefik
**Специализированные:** Triton Inference Server, Ray Serve, WebSocket, gRPC

## Чек-лист перед публикацией

- [ ] Файл создан в `app/content/ru/projects/` и `app/content/en/projects/`
- [ ] Имя файла: `{N}.{slug}.md` (N — следующий свободный номер)
- [ ] Frontmatter идентичен по структуре в обеих версиях
- [ ] `category` — только `'commercial'` или `'research'`
- [ ] `date` — формат `YYYY-MM-DD`
- [ ] `tags` — на английском языке, 8-15 штук
- [ ] `description` — не более 60 слов (помещается в 2 строки)
- [ ] `highlights` — содержат конкретные метрики (5-10 пунктов)
- [ ] Контент содержит секции: О проекте, Задача, Архитектура, Стек технологий, Ключевые достижения
- [ ] Секции используют h2 (`##`), подсекции — h3 (`###`)
- [ ] Ключевые достижения в формате: `- **Название** — описание`
- [ ] Стек технологий разбит по категориям: `**Категория:** Технология 1, Технология 2`
