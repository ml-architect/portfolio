## Обзор проекта

**Palatine Astro** -- продуктовая платформа для предоставления персональных астрологических консультаций через Telegram-бота, работающего на базе локальной языковой модели (LLM). Пользователь задает вопрос в чате Telegram, бот транслирует его в LLM через цепочку вебхуков n8n, получает ответ в режиме потоковой передачи (Server-Sent Events / JSON streaming) и отображает его в реальном времени с эффектом "печатающего" сообщения.

Архитектура состоит из трёх ключевых компонентов: **Telegram-бот** на aiogram 3, **оркестратор рабочих процессов** n8n (с PostgreSQL, pgvector для RAG и Redis для очередей), и **MCP-сервер** (Model Context Protocol) для генерации астрологических карт (натальных и синастрических) через библиотеку Kerykeion. LLM вызывается через n8n-воркфлоу, который обогащает запрос пользователя контекстом из векторной базы знаний (pgvector) и астрологическими данными из MCP-сервера.

Продукт ориентирован на B2C-аудиторию -- пользователей, интересующихся астрологией и желающих получать персонализированные прогнозы с учетом натальной карты. Бизнес-модель предусматривает подписочную монетизацию (тарифы по длительности) и пополнение пакетов сообщений.

## Стек технологий

### Backend (Telegram Bot)
- **Python 3.11** -- основной язык разработки обоих подпроектов
- **aiogram 3.22** -- асинхронный фреймворк для Telegram Bot API с поддержкой роутеров, фильтров и middleware
- **aiohttp** -- асинхронный HTTP-клиент для взаимодействия с LLM-вебхуками (streaming, FormData uploads)
- **Pydantic** -- валидация данных (транзитивно через aiogram)

### Backend (MCP-сервер)
- **FastMCP 2.10+** -- фреймворк для реализации Model Context Protocol серверов, позволяющий LLM вызывать инструменты (tools) как функции
- **Kerykeion 4.26+** -- астрологическая библиотека для вычисления натальных карт, положений планет, аспектов и генерации SVG-диаграмм
- **Pydantic (Annotated + Field)** -- типизированные описания параметров MCP-инструментов с метаданными для LLM

### Оркестрация и Workflow
- **n8n 1.106.3** -- low-code платформа автоматизации рабочих процессов, используется как middleware между ботом и LLM
- **n8n Worker** -- отдельный процесс для обработки задач из очереди (режим `EXECUTIONS_MODE=queue`)

### Базы данных
- **PostgreSQL 15** -- основное хранилище данных n8n (воркфлоу, учетные записи, история выполнений)
- **pgvector (PostgreSQL 17)** -- расширение для хранения и поиска векторных эмбеддингов; используется для RAG-пайплайна (Retrieval-Augmented Generation) -- обогащение промптов релевантными знаниями из астрологической базы
- **Redis 6** -- брокер очередей для n8n в режиме queue-based execution, health check каждые 5 секунд

### Инфраструктура и DevOps
- **Docker / Docker Compose** -- контейнеризация всех сервисов (6 контейнеров: n8n, n8n-worker, db, pgvector, redis, mcp)
- **GitLab CI/CD** -- автоматический деплой на dev и prod серверы при пуше в соответствующие ветки
- **uv (Astral)** -- современный пакетный менеджер Python, используется как в локальной разработке, так и внутри Docker-образов
- **Базовый образ ghcr.io/astral-sh/uv:python3.11-bookworm** -- оптимизированный Docker-образ с предустановленным uv

### Безопасность
- **N8N_BASIC_AUTH** -- базовая аутентификация для UI n8n
- **Docker non-root user (UID 1000)** -- понижение привилегий в контейнере бота
- **Environment-based secrets** -- все токены и пароли через переменные окружения, не зашиты в код

## Архитектура и компоненты

### 1. Telegram-бот (astro-tg-bot)
**Путь:** `Astro/astro-tg-bot/`
**Стек:** Python 3.11, aiogram 3.22, aiohttp, uv, Docker

Асинхронный Telegram-бот -- фронтенд-интерфейс платформы, обеспечивающий взаимодействие пользователей с LLM через привычный мессенджер.

- **Потоковый вывод ответов (Streaming)**: бот отправляет placeholder-сообщение ("Соединяюсь со звёздами...") и редактирует его по мере получения чанков от LLM. Реализован контроль частоты обновлений (`STREAM_UPDATE_INTERVAL = 0.2 сек`), чтобы не превышать rate limits Telegram API. Текст нормализуется и обрезается до `MAX_MESSAGE_LENGTH = 4096` символов
- **Парсинг JSON-стрима**: модуль `llm.py` реализует кастомный инкрементальный JSON-парсер (`_drain_json_stream`) на базе `json.JSONDecoder.raw_decode()`. Буфер накапливает чанки, парсер извлекает завершённые JSON-объекты типа `item` с полем `content`, обрабатывает события `error`, и сохраняет неполный буфер для следующей итерации
- **Глубокий рекурсивный парсинг ответов**: функция `_extract_text_from_data` рекурсивно обходит произвольные вложенные JSON-структуры (строки, словари, списки), ища текстовые поля по приоритетному списку ключей (`text`, `reply`, `response`, `result`, `message`, `output`, `content`). Это обеспечивает совместимость с различными форматами ответов LLM
- **Управление историей диалога**: вебхук для очистки контекста беседы (`clear_history`), позволяющий начать разговор с чистого листа
- **Административная функциональность**: система ролей на основе `ADMIN_IDS` (frozenset). Администраторы получают дополнительную кнопку "Обновить базу знаний" и могут загружать файлы через бот. Реализован stateful-механизм загрузки: `pending_uploads` (set) отслеживает, какие админы инициировали загрузку, и принимает файл только после явного запроса
- **Обновление базы знаний**: админ загружает документ через Telegram, бот скачивает файл в `BytesIO`, формирует `multipart/form-data` запрос и отправляет на вебхук n8n для обновления RAG-базы. Поддерживается произвольный MIME-тип и имя файла
- **Fallback-система**: 10 рандомизированных fallback-сообщений в астрологической тематике. При ошибке LLM или пустом ответе пользователь получает мягкое сообщение вместо технической ошибки
- **Иерархия клавиатур**: многоуровневая система `ReplyKeyboardMarkup` -- основное меню, управление подпиской (с выбором тарифа), пополнение сообщений (выбор пакета), команды. Подписка и пополнение реализованы как заглушки для будущей монетизации
- **Graceful shutdown**: регистрация обработчика `dp.shutdown` для корректного закрытия aiohttp-сессии (`llm.aclose()`)
- **Конфигурация через dataclass**: `Settings` с `slots=True` для оптимизации памяти, загрузка из env с валидацией обязательных полей и дефолтами для опциональных

### 2. n8n-сервис (n8n-service)
**Путь:** `Astro/n8n-service/`
**Стек:** n8n 1.106.3, PostgreSQL 15, pgvector (PG 17), Redis 6, FastMCP, Kerykeion, Docker Compose

Оркестрационный слой платформы -- набор Docker-сервисов, обеспечивающих обработку запросов, хранение данных и построение астрологических карт.

- **n8n как AI-оркестратор**: платформа используется для построения воркфлоу, которые принимают запрос от бота (webhook), обогащают его контекстом из векторной базы знаний, вызывают LLM и возвращают результат. Режим `EXECUTIONS_MODE=queue` с выделенным worker-процессом обеспечивает надёжную обработку очередей
- **Векторная база знаний (RAG)**: pgvector на PostgreSQL 17 хранит эмбеддинги астрологических текстов и документов. При получении вопроса пользователя n8n выполняет семантический поиск по векторной базе и добавляет релевантные фрагменты в промпт LLM. Администраторы могут обновлять базу знаний через бот, загружая новые файлы
- **MCP-сервер (Model Context Protocol)**: отдельный микросервис на FastMCP, предоставляющий LLM два инструмента:
  - `birth_chart` -- построение натальной карты по дате, времени и месту рождения. Использует Kerykeion для астрономических вычислений и генерации SVG-диаграммы, результат кодируется в base64 data URI
  - `synastry_chart` -- построение синастрической карты совместимости для двух человек с аналогичной логикой
  - Оба инструмента используют Pydantic `Annotated[..., Field(description=...)]` для типизированных описаний параметров, которые LLM видит и использует для корректного вызова
  - Транспорт: SSE (Server-Sent Events) на `0.0.0.0:8000`
  - Генерация карт на русском языке (`chart_language="RU"`)
- **Микросервисная сеть**: все контейнеры объединены в Docker bridge-сеть `internal`, сервисы общаются по именам контейнеров. Только n8n экспортирует порт наружу (10500 -> 5678)
- **Health checks**: PostgreSQL проверяется через `pg_isready` каждые 5 секунд (до 10 попыток), Redis -- через `redis-cli ping`. n8n и worker стартуют только после готовности зависимостей (`condition: service_healthy`)
- **Персистентные тома**: 4 именованных Docker volume (n8n_data, pg_n8n_data, pg_vector_n8n_data, redis_storage) для сохранения данных между перезапусками
- **Инициализация pgvector**: монтирование `schema.sql` в `/docker-entrypoint-initdb.d/` для автоматического создания схемы при первом запуске

### CI/CD

- **astro-tg-bot**: двухстадийный GitLab CI -- `deploy_dev` (ветка `dev`) и `deploy_prod` (ветка `main`). Каждая стадия использует отдельный набор переменных окружения (`DEV_*` / `PROD_*`) и свой docker-compose файл. Деплой триггерится только при изменении файлов из whitelist (YAML anchors `&file_changes`). Раздельные GitLab Runner-теги (`dev` / `prod`) для деплоя на разные серверы
- **n8n-service**: одностадийный CI -- деплой при пуше в `main`. Стратегия `down --remove-orphans` + `up -d --build` для чистого перезапуска всех контейнеров

## Ключевые достижения и фишки

- **Интеграция LLM с Model Context Protocol (MCP)**: реализован полноценный MCP-сервер на FastMCP, позволяющий языковой модели самостоятельно вызывать инструменты построения астрологических карт. LLM "видит" описания параметров через Pydantic-аннотации и автономно решает, когда и с какими данными вызвать `birth_chart` или `synastry_chart`. Это современный подход к AI-агентам, выходящий за рамки простых цепочек промптов

- **RAG-архитектура с векторным поиском**: использование pgvector для семантического поиска по астрологической базе знаний. Динамическое обновление knowledge base через Telegram-бот (администратор загружает файл -> n8n обрабатывает -> эмбеддинги сохраняются в pgvector). Это обеспечивает точные и контекстуально релевантные ответы LLM без дообучения модели

- **Потоковая передача ответов LLM в Telegram**: кастомный инкрементальный JSON-парсер, обрабатывающий поток чанков от n8n в реальном времени. Парсер использует `json.JSONDecoder.raw_decode()` для извлечения завершённых JSON-объектов из буфера, обрабатывает события `item` и `error`, и корректно работает с неполными данными в буфере. Обновление сообщения каждые 200 мс создает UX "живого" диалога

- **Рекурсивный универсальный парсинг ответов**: функция `_extract_text_from_data` обходит произвольные вложенные JSON-структуры (включая строки, содержащие JSON) и ищет текст по приоритетному списку из 7 ключей. Функция `_possible_json_slices` генерирует множество кандидатов для парсинга (полный текст, массивы, объекты, отдельные строки), обеспечивая совместимость с любым форматом ответа LLM

- **Распределённая очередь задач n8n**: режим `EXECUTIONS_MODE=queue` с Redis-брокером и выделенным worker-процессом. Это обеспечивает надежную обработку запросов под нагрузкой, изоляцию выполнения от UI n8n, и возможность горизонтального масштабирования воркеров

- **Генерация астрологических карт**: интеграция Kerykeion для вычисления положений планет, аспектов и генерации SVG-диаграмм натальных и синастрических карт. Результат кодируется в base64 data URI для передачи через MCP-протокол. Поддержка русской локализации карт

- **Production-ready Docker-инфраструктура**: 6 контейнеров с health checks, dependency ordering, persistent volumes, bridge-сетью, non-root user в контейнере бота. Оптимизация Docker-образа бота: отдельный слой для зависимостей (`uv sync --no-dev`), компиляция bytecode (`UV_COMPILE_BYTECODE=1`), copy-режим линковки (`UV_LINK_MODE=copy`)

- **Отказоустойчивость на всех уровнях**: fallback-сообщения при недоступности LLM, обработка `TelegramBadRequest` (message not modified, message too long), graceful shutdown с закрытием HTTP-сессий, конфигурация `restart: unless-stopped` / `restart: always` для всех контейнеров

- **Двухсредовый CI/CD с селективным триггерингом**: GitLab CI с YAML anchors для DRY-конфигурации, раздельные переменные окружения и runner-теги для dev/prod, whitelist файлов для триггеринга деплоя (предотвращение лишних пересборок при изменении README и т.п.)
