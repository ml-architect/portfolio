## О проекте

Palatine Astro — продуктовая AI-платформа для персональных астрологических консультаций через Telegram. Пользователь задаёт вопрос в чате, бот передаёт запрос в LLM через цепочку n8n-воркфлоу, обогащает контекстом из векторной базы знаний (pgvector) и астрологическими данными из MCP-сервера, и отображает ответ в реальном времени с эффектом «печатающего» сообщения.

Платформа ориентирована на B2C-аудиторию с подписочной монетизацией. Архитектура из трёх компонентов: Telegram-бот, оркестратор n8n и MCP-сервер для генерации астрологических карт.

## Задача

Создать AI-платформу, в которой LLM не просто отвечает на вопросы, а автономно принимает решения о вызове инструментов — построении натальных и синастрических карт. Ответы должны быть обогащены контекстом из базы знаний (RAG) и отображаться в реальном времени. Администраторы должны иметь возможность обновлять базу знаний прямо через бот.

## Архитектура

### Telegram-бот (astro-tg-bot)

Асинхронный бот на aiogram 3 с потоковым выводом ответов LLM. Отправляет placeholder-сообщение и редактирует его по мере получения чанков с интервалом 200 мс. Кастомный инкрементальный JSON-парсер на `json.JSONDecoder.raw_decode()` извлекает завершённые JSON-объекты из буфера потока. Рекурсивный парсер `_extract_text_from_data` обходит произвольные вложенные JSON-структуры, обеспечивая совместимость с любым форматом ответа LLM.

Административный модуль: ролевая модель на `ADMIN_IDS`, stateful-механизм загрузки файлов для обновления базы знаний через бот → multipart/form-data → n8n → pgvector.

### MCP-сервер (Model Context Protocol)

Микросервис на **FastMCP**, предоставляющий LLM два инструмента:
- `birth_chart` — натальная карта по дате, времени и месту рождения
- `synastry_chart` — карта совместимости для двух человек

Используется **Kerykeion** для астрономических вычислений и генерации SVG-диаграмм. Параметры описаны через `Pydantic Annotated + Field(description=...)` — LLM «видит» описания и автономно решает, когда вызвать инструмент. Транспорт: SSE на порту 8000.

### n8n-оркестрация

Middleware между ботом и LLM. Режим `EXECUTIONS_MODE=queue` с Redis-брокером и выделенным worker-процессом. pgvector на PostgreSQL 17 хранит эмбеддинги для RAG — семантический поиск релевантных фрагментов при каждом запросе. 6 Docker-контейнеров с health checks, dependency ordering и persistent volumes.

## Стек технологий

**Backend:** Python 3.11, aiogram 3.22, aiohttp (streaming), FastMCP 2.10+, Kerykeion (астрономия + SVG), Pydantic

**AI/ML:** n8n (AI-оркестрация), pgvector (RAG, семантический поиск), MCP (Model Context Protocol)

**Данные:** PostgreSQL 15, pgvector (PostgreSQL 17), Redis 6

**Инфраструктура:** Docker Compose (6 сервисов), GitLab CI/CD (dev/prod), uv (Astral), non-root Docker runtime

## Ключевые достижения

- **Model Context Protocol (MCP)** — LLM автономно вызывает инструменты построения астрологических карт через типизированные Pydantic-аннотации. Современный подход к AI-агентам, выходящий за рамки простых цепочек промптов
- **RAG с динамическим обновлением** — pgvector для семантического поиска по базе знаний с возможностью обновления через Telegram-бот без перезапуска сервисов
- **Потоковый вывод LLM в Telegram** — кастомный инкрементальный JSON-парсер для обработки потока чанков в реальном времени с контролем rate limits API
- **Рекурсивный универсальный парсинг** — обход произвольных вложенных JSON-структур по приоритетному списку из 7 ключей, совместимость с любым форматом ответа LLM
- **Queue-based n8n** — распределённая обработка с Redis-брокером и выделенным worker, обеспечивающая надёжность под нагрузкой
- **Production Docker-инфраструктура** — 6 контейнеров с health checks, multi-layer caching (uv), bytecode compilation, non-root runtime, двухсредовый CI/CD с селективным триггерингом
